## Scope — 4 polish items

All 4 surfaced during T3 LIGHT live verification on 2026-05-18.

### Item 1: `dispatch-log.jsonl` silent on no-dispatch coordinator decisions

**Observed**: `aiplus agent route "describe the current git status in one sentence"` → coordinator scored complexity=1 risk=0.15 → tier LIGHT_NO_CODE → `Staffing roles: []` → CEO handles directly → `dispatch-log.jsonl` got ZERO new lines.

**Why it matters**: 2-week Owner dogfood needs full audit trail of every coordinator decision, not only ones that dispatched roles. Otherwise Owner cannot retroactively answer "which tasks were LIGHT_NO_CODE this week" or "did the scorer mis-tier anything I had to retry".

**Fix**: emit a `coordinator_decision` event in `dispatch-log.jsonl` for EVERY scored task, including no-dispatch outcomes.

**Schema (CEO designs final form; below is starting suggestion)**:
```jsonl
{"ts":"2026-05-18T14:23:01Z","event":"coordinator_decision","task_excerpt":"describe the current git status","complexity":1,"risk":0.15,"tier":"LIGHT_NO_CODE","code_change":false,"design_impact":false,"consultant":"skip","staffing_roles":[],"dispatched":false}
```

Then for actual role dispatches, the existing per-role log lines remain unchanged (they get the additional event-pairing context via shared `task_id` field if CEO wants to wire that).

**`task_excerpt`** truncate at 200 chars + ellipsis to avoid logging entire prompts. Add to redaction guard (same patterns as `cache.rs`).

### Item 2: `aiplus doctor` silent on BWS-backed-secret + no env-key state

**Observed**: When Owner's backend is BWS (per auto-memory `bws_setup.md`) and Owner runs `aiplus agent route "..."` directly without `aiplus secret-broker run --aliases <provider> --` wrapper, the adapter spawns runtime → runtime tries to read `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` from env → not present → adapter fails with auth error inside the spawned process.

`aiplus doctor` today does NOT detect this preventable failure mode.

**Why it matters**: BWS is Owner's actual setup. This is the #1 onboarding cliff for any non-Builder user who reads "just run `aiplus agent route ...`" in the README.

**Fix**: add a `doctor` check `CHECK_SECRET_BROKER_RUNTIME_AUTH`:
- If `aiplus secret-broker status` reports BWS backend present AND
- agent-team has at least 1 active role configured AND
- one of `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` is NOT in current process env
- Then emit WARN (NOT FAIL) with copy-pasteable fix:
  ```
  WARN_SECRET_BROKER_RUNTIME_AUTH
  Detected BWS backend + active agent-team roles, but no provider key in env.
  Adapter dispatch will fail with auth error.
  Fix: wrap your dispatch with
    aiplus secret-broker run --aliases anthropic,openai -- aiplus agent route "<task>"
  Or set ANTHROPIC_API_KEY / OPENAI_API_KEY in your shell once.
  ```

WARN (not FAIL) because keyring-backend users + Builder-mode users who export env vars elsewhere are unaffected.

**Also**: add a one-line mention in `aiplus agent route --help` and in `aiplus agent route`'s FIRST-RUN output (one-time hint, gated by absence of `~/.config/aiplus/.route_first_run_seen` marker).

### Item 3: Scorer calibration drift undetected

**Observed**: Advisor predicted `describe git status` → risk=0.0. Actual: risk=0.15. Behavior was still correct (tier=LIGHT_NO_CODE unchanged). But this is a small drift signal — scorer rubric has implicit baselines that aren't documented or regression-tested.

**Why it matters**: When v0.3.1 P1 (expert auto-summoning, risk-based forced summoning, TTL honoring) lands, scorer logic gets touched. Without a calibration test, drift goes undetected until Owner notices over-staffing or under-staffing in real workflow.

**Fix**: add `crates/aiplus-cli/tests/coordinator_calibration.rs` driven by a TOML fixture:

`crates/aiplus-cli/tests/fixtures/coordinator_calibration.toml`:
```toml
# Each entry asserts coordinator scoring matches expected band.
# Risk uses _max not exact: scorer is deterministic but rubric may evolve;
# bands prevent over-rigid tests while still catching regressions.

[[task]]
input = "describe the current git status in one sentence"
expected_complexity = 1
expected_risk_max = 0.25
expected_tier = "LIGHT_NO_CODE"
notes = "T3 LIGHT live verification baseline"

# CEO designs the rest — target ~15-20 entries spanning LIGHT_NO_CODE,
# LIGHT_CODE, MEDIUM, HEAVY, with at least 3 entries per tier and
# at least 2 boundary cases (complexity 2↔3, risk 0.69↔0.70).
```

Plus a Rust test harness `tests/coordinator_calibration.rs` that loads the fixture, calls `coordinator::score()` for each entry, and asserts `(complexity, tier)` exactly + `risk <= expected_risk_max`.

This test is a regression net for future scorer changes, NOT a re-derivation of the rubric. Fixture entries come from prior real tasks (you can pull historical examples from Owner's git activity over the last week + DESIGN.md §9 / §17 examples).

### Item 4: No `--score-only` mode on `aiplus agent route`

**Observed**: There is no way to ask coordinator "what would you score this task?" without actually dispatching. Owner cannot pre-flight a task. The calibration test in Item 3 also benefits from this flag (test can drive scorer via subprocess CLI rather than only via Rust API).

**Fix**: add `aiplus agent route --score-only "<task>"`:
- Runs coordinator scorer + tier classifier
- Prints same `Adaptive coordinator: complexity=N risk=F tier=T code_change=B design_impact=B consultant=B` line to stdout
- Also prints `Plan step: <what would happen>` and `Would staff: [<roles>]` (dry-run plan)
- Does NOT call consultant, does NOT staff roles, does NOT spawn adapters, does NOT write cache
- DOES write a `coordinator_decision` event to dispatch-log with `dispatched=false, mode=score_only` (per Item 1)
- Exit 0

Add Chinese alias `--打分` (consistent with existing CLI Chinese aliases).
