## Scope — 3 P1 features

All 3 specced in `v0.3-adaptive-coordinator.md` as P1 #5, #6, #7. Implementation order below is by isolation + risk (smallest blast radius first).

### Feature 1: TTL honoring (D3 in goal doc)

**Status today**: `warm_bench_ttl_seconds` field exists in role TOML (default 1800), is read into config, but is NOT consulted at dispatch time. Cache loads regardless of age.

**What to ship**:
1. At dispatch in `route.rs` adaptive path: before calling `cache::load_disk_snapshot`, check `now - cache_meta.last_used_at > ttl_seconds`. If yes → invalidate (mark cold-start) + log `ttl_expired=true`.
2. New `coordinator_decision` field: `ttl_expired: bool` (always present for dispatched events; null for `--score-only`).
3. New cache config gate: `[cache] enforce_ttl = false` (default; today's behavior). `enforce_ttl = true` enables P1 enforcement.
4. `aiplus doctor` adds INFO line (not WARN): `cache_age=Xs ttl=Ys role=<r>` for each cached role; WARN only when `enforce_ttl=true` AND age > TTL.
5. Unit tests in `tests/ttl_invalidation_smoke.rs`: stale cache → cold-start; fresh cache → warm hit; enforce_ttl=false → ignore TTL (regression safety).

**Why first**: isolated to `cache.rs` + small `route.rs` + `doctor.rs` touchpoint. Smallest blast radius. Default OFF (`enforce_ttl=false`) means existing dogfood unaffected until Owner opts in.

### Feature 2: Risk-based forced summoning (D2)

**Status today**: `staffing_roles()` in `coordinator.rs` returns fixed list per tier. Risk score is computed but ignored after tier classification.

**What to ship**:
1. In `coordinator.rs`: after tier-baseline staffing computed, post-process staffing list:
   - If `risk >= 0.7`: ensure `reviewer` in staffing (add if missing)
   - If `risk >= 0.85`: also ensure `qa` in staffing (add if missing)
2. Dedup applies: HEAVY already has both → no-op.
3. New `coordinator_decision` field: `forced_by_risk: ["reviewer"]` or `[]`. Captures which roles came from this gate (vs tier-baseline).
4. `--score-only` output line: append `(forced by risk: reviewer)` annotation when applicable.
5. Unit tests: 4 thresholds × 4 tiers = 16-cell matrix in `tests/risk_forced_smoke.rs` covering all combinations.
6. Calibration fixture: add 4-6 entries crossing risk thresholds at LIGHT_CODE tier (force trips) and HEAVY tier (no-op).

**Why second**: touches only `coordinator.rs` staffing logic + `route.rs` log emit. Doesn't change schema (just adds a field).

### Feature 3: Expert auto-summoning (D1)

**Status today**: experts exist as role TOML files but coordinator doesn't reference them. Staffing is fixed templates only.

**What to ship**:
1. **Role TOML schema extension** — optional new section:
   ```toml
   [autosummon]
   keywords = ["支付", "payment", "secure", "auth", "credentials"]
   match_mode = "any"   # "any" = OR; "all" = AND (latter rare)
   priority = 100        # higher = more important; cap selection if cluster too large
   ```
   Absent section → role does NOT auto-summon. Backward compatible.
2. Coordinator builds a registry from all role TOMLs with `[autosummon]` at startup.
3. After tier-baseline + risk-forced staffing, scan task text for keyword matches:
   - For each matched role, add to staffing.
   - Dedup. Cap final cluster at `tier_baseline_size + 2` (so LIGHT_NO_CODE can grow from 0 → ≤ 2 experts).
4. New `coordinator_decision` field: `auto_summoned: ["security-expert"]` or `[]`.
5. `--score-only` output: append `(auto-summoned: security-expert ['支付'])` showing which keyword matched.
6. **Initial trigger sets** — ship trigger TOML for ≥3 of 6 v0.1 experts (per goal doc AC):
   - `security-reviewer`: keywords `["支付", "payment", "auth", "secret", "credentials", "vulnerability", "OWASP", "CVE"]`
   - `tech-writer`: keywords `["README", "doc", "documentation", "tutorial", "API docs"]`
   - `ai-integration-specialist`: keywords `["LLM", "prompt", "embedding", "RAG", "fine-tune", "ANTHROPIC_API_KEY"]`
   (CEO designs final lists based on each expert's persona; above is starting point.)
7. Unit + integration tests: `tests/auto_summon_smoke.rs` — D5 task `实现支付接口` → security-reviewer added; `describe git status` → no auto-summon; multi-match task → multi-expert + dedup.
8. Calibration fixture: 4-6 new entries covering auto-summon + no-auto-summon + multi-match.

**Why last**: largest blast radius — new schema field across multiple TOML files + new registry building in coordinator + new parser. Best done after #1 + #2 are stable.
