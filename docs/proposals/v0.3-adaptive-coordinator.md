## TL;DR

CONTRACT v1 FROZEN means the adapter layer is production-ready: CEO can spawn role agents in any of 3 runtimes (Claude Code / OpenCode / Codex) under uniform contract. **But CEO's task routing is still rule-based simple — it does not yet score tasks, scale staffing, or auto-summon experts.** v0.3 implements the adaptive coordinator inside CEO so the team works as DESIGN.md §5 envisioned.

## Goals (D1-D5, Owner-signed when approved)

- **D1**: CEO scores every dispatched task on **complexity** (1-5) and **risk** (0.0-1.0) using `aiplus-auto-team-consultant`'s existing algorithm (re-used, NOT re-invented).
- **D2**: Default LIGHT / MEDIUM / HEAVY staffing scales applied automatically (see §9.2 table).
- **D3**: Expert auto-summoning when task keywords match expert trigger conditions (see §5.3 table).
- **D4**: Per-agent warm-bench TTL config honored from `.aiplus/agents/<role>.toml`.
- **D5**: Acceptance per §17 v0.3: Owner says "实现支付接口" → CEO scores 4/0.85 → fires consultant lenses → staffs Engineer A + Reviewer + Security Reviewer → all run in their own worktrees → CEO integrates and reports back without Owner intervention.

## Non-goals

- No new scoring algorithm — reuse `aiplus-auto-team-consultant`'s Coordinator.
- No new CLI surface — same `aiplus agent route <task>` entry point; behavior changes underneath.
- No cross-runtime parallelism (v0.4 scope).
- No new role personas or expert directory changes.

## Scope items

### P0 — must do this milestone

1. **CEO scorer**: implement task scoring (complexity 1-5 + risk 0.0-1.0) by calling `aiplus-auto-team-consultant`'s public scoring API; cache scores keyed by task hash.
2. **Tier classifier**: map score → LIGHT / MEDIUM / HEAVY per §9.2 table.
3. **Staffing dispatcher**: per tier, dispatch roles via adapter layer (CONTRACT v1) in the order defined by §9.2.
4. **Two-step CEO flow (§9.3)**: for MEDIUM + HEAVY, fire consultant first (read-only plan-time advice), then staff. LIGHT bypasses consultant for speed.

### P1 — should do

5. **Expert auto-summoning**: scan task text for expert-trigger keywords (§5.3 table); add matched experts to active team via existing `aiplus agent invite` machinery.
6. **Risk-based forced summoning**: risk ≥ 0.7 always pulls Security Reviewer regardless of tier (§9.2 footnote).
7. **Per-agent TTL honoring**: warm-bench TTL loaded from `.aiplus/agents/<role>.toml`; cache invalidation per §6.5 triggers.

### P2 — nice to have

8. **Score audit trail**: append scoring decisions to team memory for retrospective.
9. **Manual score override**: `aiplus agent route --tier HEAVY <task>` to force a tier.
10. **Consultant-fired-but-team-disagreed signal**: if staff produces output that contradicts consultant lens findings, surface in CEO's report.

## Phases

- **Phase A** (P0): scorer + tier classifier + staffing dispatcher + two-step flow. ~1 week.
- **Phase B** (P1): expert auto-summoning + risk-based forced + TTL honoring. ~3-5 days.
- **Phase C** (P2): polish. Opportunistic.

## Acceptance criteria

- D5 scenario passes end-to-end without Owner intervention.
- LIGHT tasks (e.g., status checks) bypass consultant + complete < 30s.
- HEAVY tasks (complexity ≥ 5 OR risk ≥ 0.7) fire consultant + staff full team + integrate.
- Auto-summoning works for at least 3 of the 6 v0.1-shipped experts (AI Integration, Security Reviewer, Tech Writer).
- Per-role TTLs from .toml override defaults verifiably (via `aiplus agent status --verbose`).

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Scorer over-estimates complexity → over-staffs LIGHT tasks → wasteful | Calibration test: 20 historical Owner tasks scored manually, compare to scorer output; tune thresholds |
| Expert trigger keywords match too broadly → expert summoned unnecessarily | Triggers require keyword AND context match (not single-word); doc per-expert false-positive rate |
| Auto-summoning fires for confidential/restricted task → wrong expert sees task text | Trigger evaluation happens BEFORE memory injection; expert only sees task text it would have seen via manual `agent invite` |
| Two-step flow doubles latency for MEDIUM tasks | Acceptable; consultant lens is ~10-20s, far cheaper than rework from missed risk |
| Re-use of consultant Coordinator algorithm couples plugins | Already required per DESIGN.md §4 + §9.1; this is intentional consolidation, not new coupling |
