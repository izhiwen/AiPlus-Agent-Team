# v0.2 — Disk-Warm Cache (Opt-in Option B per DESIGN.md §6.3)

**Status**: DRAFT — Owner approval pending.
**Drafted by**: Advisor (post-CONTRACT v1 FROZEN)
**Date**: 2026-05-18
**Authoritative reference**: `DESIGN.md` §6.3 (Warm Bench under an ephemeral CLI), §16 (STOP Gates)

## TL;DR

CONTRACT v1 enabled adapter spawning across runtimes. Each spawn currently re-loads role state (persona + memory + workspace cursor) from disk → ~2-5s cold start. Disk-warm cache persists state snapshots under `~/.cache/aiplus-agent-team/<project>/<role>.cbor` with TTL, so subsequent `aiplus agent route` calls within the TTL window resume instantly. Closes v0.2's last roadmap item per DESIGN.md §17.

## Goals (D1-D4, Owner-signed when approved)

- **D1**: Cross-CLI warm bench — same role dispatched twice within TTL = 2nd dispatch sees `cache_source=disk_warm` in `aiplus agent status --verbose`.
- **D2**: Opt-in only — feature disabled by default; enabled via explicit `aiplus agent cache --enable-disk` per DESIGN.md §16.
- **D3**: Cache file integrity — checksum-verified on read; corruption → cold-start, never silent bad-state continuation (mirrors §4.3 mapping corruption policy).
- **D4**: Privacy boundary honored — cache contents pass through `aiplus-agent-memory` redaction pipeline before disk write (mirrors CONTRACT §9 trace requirement).

## Non-goals

- No cross-machine cache sync (single host only).
- No automatic re-warming after Owner edits to `.aiplus/agents/<role>.toml` (existing §6.5 invalidation triggers apply).
- No cache for owner-facing roles (Advisor + CEO) — they're conversation-style, low cold-start cost.
- No replacement of v0.1 Option A (in-process cache) — Option B layers on top.

## Design

### Path layout

```
~/.cache/aiplus-agent-team/
└── <project_basename>/
    ├── <role>.cbor                  # binary snapshot
    ├── <role>.cbor.sha256           # integrity check
    └── _cache_meta.json             # TTL, last_used_at, schema_version
```

Project keyed by directory basename (matches §10.1 `worktree_path` template).

### Cache lifecycle

1. **Write**: after role completes a task, adapter (via CEO) writes snapshot if `cache.enable_disk=true` in `.aiplus/agent-team.toml`.
2. **Read**: at next dispatch of same role, CEO checks `_cache_meta.json` for TTL validity + sha256 match → loads instantly. Miss → cold-start from `.aiplus/agents/<role>.toml`.
3. **Invalidate**: all §6.5 triggers fire (memory.add, branch commit, integrate, persona edit, etc.). Plus new trigger: cache file age exceeds TTL.
4. **Purge**: `aiplus agent cache --clear` removes entire `~/.cache/aiplus-agent-team/<project>/` for current project.

### Snapshot contents

- Persona (resolved from `.aiplus/agents/<role>.toml` + `personas/<role>.md`)
- Memory bundle (3-layer merged, post-redaction)
- Workspace cursor (current worktree HEAD + dirty-flag)
- Adapter session-mapping reference (per CONTRACT §4 mapping file)

Excluded from snapshot:
- Raw transcripts (kept in `.aiplus/agent-memory/<role>/`, not duplicated)
- Secrets / API keys (env-only per CONTRACT §6.1, never serialized)
- Tool-output payloads (re-fetched on demand)

### TTL defaults

Match DESIGN.md §6.4 per-role TTLs (Advisor 60min, CEO 60min, etc.). Disk cache reuses same TTL — no second TTL concept to confuse Owner.

## Acceptance criteria

- AC1: enable feature → run `aiplus agent route engineer-a "..."` → exit CLI → re-run same role within TTL → `aiplus agent status --verbose` shows `cache_source=disk_warm`.
- AC2: disable feature → no `~/.cache/aiplus-agent-team/` directory created.
- AC3: manually corrupt `<role>.cbor` → next dispatch reports cold-start + surfaces warning in `aiplus agent doctor`.
- AC4: cache file inspected via `cat ~/.cache/aiplus-agent-team/<project>/<role>.cbor | strings` shows no secrets / API keys / raw passwords.
- AC5: `aiplus agent cache --clear` removes only current project's cache, not other projects'.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Cache content sensitivity (memory bundle snapshots) | Redaction-on-write; opt-in only; `--clear` escape hatch |
| Disk space growth | Cache per role per project capped at 5MB; LRU eviction when project total > 50MB |
| Stale cache after Owner edit | §6.5 invalidation triggers; mtime check on `.aiplus/agents/<role>.toml` against cache write time |
| Cross-project cache pollution | Project basename in path; no cross-project read |
| `~/.cache/` on iCloud/Dropbox | Doctor check: detect sync path under cache dir → warn Owner; recommend `--clear` and re-enable |

## Implementation phases

- **Phase A**: snapshot read/write primitive + sha256 verify. ~2-3 days.
- **Phase B**: CEO integration (cache check before dispatch). ~2 days.
- **Phase C**: `aiplus agent cache` subcommand (enable / disable / clear / status). ~1 day.
- **Phase D**: doctor checks + iCloud/Dropbox warning. ~1 day.
- **Phase E**: dogfood + AC1-AC5 verify. ~2 days.

Total: ~1.5 weeks.

## Owner-gates

- `aiplus agent cache --enable-disk` is explicit consent step per DESIGN.md §16. Feature stays OFF until Owner runs this.
- Owner can disable any time via `aiplus agent cache --disable-disk`; existing cache files retained until `--clear`.

## Out of scope (v0.2.x patch territory)

- Multi-machine cache sync via Bitwarden Secrets Manager
- Cache compression (cbor is already compact)
- Selective per-role disk-cache enable (all-or-nothing in v0.2 per DESIGN.md §6.3 Option B)
